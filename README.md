# CeaAI MVP Documentation

## Быстрый запуск технического MVP

В этом репозитории реализован Telegram-бот CeaAI с SQLite/Postgres, mock-платежами для локальной разработки, YooKassa-платежами для прода, реальными AI-провайдерами DeepSeek/OpenAI для текста и OpenAI GPT Image для картинок, а также mock-заглушками для остальных типов генераций.

### Что внутри

- `ceai/bot` — команды, меню и callback handlers Telegram-бота.
- `ceai/services` — бизнес-логика пользователей, подписок, платежей, коинов и генераций.
- `ceai/repositories` — доступ к SQLite-таблицам из модели данных.
- `ceai/providers` — реальные провайдеры DeepSeek/OpenAI для текста, OpenAI GPT Image для картинок и mock-заглушки video/tts.
- `migrations/001_init.sql` — схема БД из `Документация.md`.
- `migrations/postgres/001_init.sql` — та же схема для Postgres.
- `ceai/seed.py` — стартовые тарифы и цены моделей.
- `tests/test_business_logic.py` — минимальная проверка ключевых правил.
- `Dockerfile` — production-образ для Railway/Docker-хостинга.
- `railway.json` — настройки Railway build/deploy и healthcheck.

### Локальный запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e .
cp .env.example .env
```

В `.env` нужно указать токен Telegram-бота:

```bash
TELEGRAM_BOT_TOKEN=put-your-telegram-bot-token-here
DATABASE_URL=sqlite:///./data/ceai.sqlite3
MOCK_PAYMENT_BASE_URL=https://mock-payments.local/pay
PAYMENT_PROVIDER=mock
AI_PROVIDER_MODE=auto
DEEPSEEK_API_KEY=your-deepseek-api-key
OPENAI_API_KEY=your-openai-api-key
OPENAI_IMAGE_API_KEY=your-openai-image-api-key
```

`OPENAI_API_KEY` используется для ChatGPT-текста, `OPENAI_IMAGE_API_KEY` — для GPT Image. Если отдельный ключ для картинок не задан, бот использует обычный `OPENAI_API_KEY`. `AI_PROVIDER_MODE=auto` означает: DeepSeek/OpenAI работают через реальные API при наличии ключей, а неподключенные типы генераций остаются на mock-заглушках. OpenAI GPT Image без ключа не отдаёт mock-картинку, а завершает генерацию ошибкой с возвратом коинов. Для полностью тестового режима поставьте `AI_PROVIDER_MODE=mock`.

Создать БД и заполнить тарифы/модели можно отдельно:

```bash
python -m ceai.seed
```

Запуск бота:

```bash
python -m ceai.main
```

При старте бот также автоматически применяет миграции и seed-данные.

### Деплой на Railway

Railway использует `Dockerfile` и `railway.json`. В проде бот работает через Telegram webhook, поэтому у сервиса должен быть публичный домен.

1. Создайте проект Railway из GitHub-репозитория.
2. В сервисе откройте `Settings -> Networking` и нажмите `Generate Domain`.
3. Добавьте Postgres в Railway и подключите `DATABASE_URL` к сервису. В продакшене SQLite запрещён по умолчанию: файл внутри контейнера может быть пересоздан при деплое, и тогда пропадут пользователи, подписки, платежи и балансы.
4. Укажите переменные окружения:

```bash
TELEGRAM_BOT_TOKEN=your-botfather-token
DATABASE_URL=postgresql://user:password@host:5432/dbname
APP_ENV=production
MOCK_PAYMENT_BASE_URL=https://mock-payments.local/pay
PAYMENT_PROVIDER=yookassa
YOOKASSA_SHOP_ID=your-yookassa-shop-id
YOOKASSA_SECRET_KEY=your-yookassa-secret-key
YOOKASSA_API_BASE_URL=https://api.yookassa.ru/v3
YOOKASSA_WEBHOOK_PATH=/payments/yookassa/webhook
YOOKASSA_RETURN_PATH=/payments/yookassa/return
TELEGRAM_STARS_AMOUNT=0
AI_PROVIDER_MODE=auto
DEEPSEEK_API_KEY=your-deepseek-api-key
OPENAI_API_KEY=your-openai-api-key
OPENAI_IMAGE_API_KEY=your-openai-image-api-key
ADMIN_TELEGRAM_USERNAMES=samescam
SUPPORT_USERNAME=cea_help
```

`TELEGRAM_STARS_AMOUNT` управляет суммой Telegram Stars в счёте. По умолчанию `1`:
любой тариф будет выставлен как 1 Star. Для нормального прода поменяйте эту переменную
на нужную сумму в Railway и перезапустите сервис.

`APP_BASE_URL` можно не задавать: Railway сам отдаёт `RAILWAY_PUBLIC_DOMAIN`, а приложение превращает его в `https://...` и ставит webhook. Если нужно указать домен вручную:

```bash
APP_BASE_URL=https://your-service.up.railway.app
TELEGRAM_WEBHOOK_PATH=/telegram/webhook
TELEGRAM_WEBHOOK_SECRET=change-me
```

5. После деплоя проверьте:

```text
https://your-service.up.railway.app/healthz
https://your-service.up.railway.app/telegram/status
https://your-service.up.railway.app/payments/yookassa/return
```

В `/telegram/status` видно, какой webhook реально установлен в Telegram и есть ли последняя ошибка доставки.

### Реальная оплата через YooKassa

Для реальной оплаты используется redirect-сценарий YooKassa:

```text
Бот создает payment в YooKassa
-> пользователь открывает confirmation_url
-> YooKassa присылает webhook payment.succeeded
-> backend проверяет payment через YooKassa API
-> backend начисляет коины один раз
```

Что нужно указать на хостинге:

```bash
PAYMENT_PROVIDER=yookassa
APP_BASE_URL=https://your-service.up.railway.app
YOOKASSA_SHOP_ID=your-shop-id
YOOKASSA_SECRET_KEY=your-secret-key
YOOKASSA_WEBHOOK_PATH=/payments/yookassa/webhook
YOOKASSA_RETURN_PATH=/payments/yookassa/return
```

В личном кабинете YooKassa добавьте входящее уведомление:

```text
Event: payment.succeeded
URL: https://your-service.up.railway.app/payments/yookassa/webhook
```

`/payments/yookassa/return` нужен только как страница возврата пользователя после оплаты. Коины там не начисляются; начисление идет только после webhook и проверки статуса платежа через API YooKassa.

### Оплата криптой через Crypto Pay

Для testnet используйте токен приложения из `@CryptoTestnetBot`.
Для прода используйте `@CryptoBot` и замените API base на `https://pay.crypt.bot/api`.

```bash
CRYPTO_PAY_TOKEN=your-crypto-pay-app-token
CRYPTO_PAY_API_BASE=https://testnet-pay.crypt.bot/api
CRYPTO_PAY_WEBHOOK_SECRET=change-this-long-random-secret
CRYPTO_PAY_WEBHOOK_PATH=/payments/crypto/webhook
CRYPTO_PAY_ACCEPTED_ASSETS=USDT
```

Webhook URL в Crypto Pay:

```text
https://your-service.up.railway.app/payments/crypto/webhook/change-this-long-random-secret
```

Сценарий:

```text
Бот создает invoice в Crypto Pay
-> пользователь открывает pay_url
-> Crypto Pay присылает webhook invoice_paid
-> backend проверяет подпись Crypto-Pay-API-Signature
-> backend начисляет коины один раз
```

### Проверка сценария в Telegram

```text
/start
-> Тарифы
-> выбрать Старт / Базовый / Про
-> Перейти к оплате
-> AI-инструменты
-> выбрать модель
-> отправить prompt
-> получить mock result
-> Баланс / История
```

Чтобы проверить возврат коинов при ошибке AI-провайдера, выберите модель и отправьте prompt `mock_error`.

### Тесты бизнес-логики

```bash
python -m unittest discover -s tests -v
```

Проверяется:

- успешная тестовая оплата начисляет коины один раз;
- повторный webhook не начисляет коины второй раз;
- успешная генерация списывает коины;
- ошибка генерации возвращает коины;
- нельзя генерировать без активной подписки;
- нельзя генерировать при нехватке коинов;
- реферальная ссылка привязывает приглашенного пользователя;
- успешная оплата приглашенного начисляет 30% один раз.

## Суть за 1 минуту

CeaAI - это Telegram-бот, где пользователь покупает подписку, получает коины и тратит их на разные AI-инструменты: чат с нейросетями, генерацию картинок и видео, озвучку текста и работу с контентом.

MVP должен проверить не "можем ли мы подключить много нейросетей", а более важный вопрос: готовы ли пользователи платить за удобный AI-агрегатор прямо в Telegram.

Базовая логика:

```text
Пользователь покупает подписку
    -> получает коины
    -> выбирает AI-инструмент
    -> отправляет запрос
    -> backend проверяет баланс
    -> вызывает нужную нейросеть
    -> бот отправляет результат
```

Главные части MVP:

- Telegram-бот как основной интерфейс;
- backend, где живет вся бизнес-логика;
- база данных для пользователей, подписок, платежей и генераций;
- внешняя платежная система со ссылкой на оплату;
- внутренняя валюта коины;
- адаптеры для подключения разных AI-провайдеров.

## 1. Что такое CeaAI

CeaAI - это Telegram-бот-агрегатор нейросетей. Пользователь покупает подписку, получает внутренние коины/лимиты и тратит их на разные AI-инструменты:

- общение с нейросетями: ChatGPT, Claude, DeepSeek и другие;
- генерация изображений;
- генерация видео;
- озвучка текста;
- обработка и улучшение текста;
- будущие инструменты для маркетинга, SEO, контента и бизнеса.

Главная идея продукта: пользователь не разбирается в разных AI-сервисах, API, VPN, оплатах и подписках. Он открывает одного Telegram-бота, выбирает задачу, отправляет запрос и получает результат.

## 2. Формат продукта

На старте CeaAI делается именно как Telegram-бот, а не как Mini App.

Почему бот:

- быстрее сделать MVP;
- пользователю привычнее писать запросы в чат;
- проще отправлять текст, фото, видео и документы;
- легче тестировать спрос;
- меньше затрат на frontend.

Mini App можно добавить позже как дополнительный интерфейс для:

- красивой галереи генераций;
- истории работ;
- личного кабинета;
- управления подпиской;
- сложных настроек генерации.

## 3. Основной пользовательский сценарий

1. Пользователь открывает Telegram-бота.
2. Нажимает `Start`.
3. Бот показывает приветствие и главное меню.
4. Пользователь видит:
   - баланс коинов;
   - активную подписку;
   - доступные AI-инструменты;
   - кнопку пополнения/покупки подписки.
5. Пользователь покупает подписку через внешнюю платежную ссылку.
6. После успешной оплаты бот начисляет коины.
7. Пользователь выбирает нейросеть или задачу.
8. Бот показывает стоимость действия в коинах.
9. Пользователь отправляет запрос.
10. Backend проверяет подписку и баланс.
11. Backend отправляет запрос в нужный AI-провайдер.
12. Пользователь получает результат в Telegram.
13. С баланса списываются коины.

## 4. Подписки и коины

В CeaAI используется гибридная модель:

- пользователь покупает подписку;
- подписка выдает определенное количество коинов на период;
- коины тратятся на AI-действия;
- если коины закончились, пользователь может докупить пакет коинов или перейти на тариф выше.

Коины нужны, чтобы:

- гибко оценивать разные нейросети;
- не объяснять пользователю себестоимость API;
- контролировать расходы;
- менять цены моделей без изменения логики подписок.

## 5. Черновая тарифная сетка

Точные лимиты нужно будет пересчитать после выбора моделей и понимания себестоимости API.

### Start

Пример: 300 рублей в месяц.

Для пользователей, которые хотят попробовать сервис.

Может включать:

- небольшой пакет коинов;
- доступ к текстовым нейросетям;
- ограниченное количество генераций изображений;
- без приоритета в очереди.

### Plus

Пример: 800 рублей в месяц.

Основной тариф, который должен быть самым выгодным и популярным.

Может включать:

- больше коинов;
- общение с ChatGPT, Claude, DeepSeek;
- генерацию изображений;
- ограниченную генерацию видео и озвучку текста;
- историю генераций.

### Pro

Пример: 3000 рублей за 2 месяца.

Тариф для активных пользователей, создателей контента, маркетологов и владельцев небольшого бизнеса.

Может включать:

- большой пакет коинов;
- доступ к дорогим моделям;
- больше генераций видео и озвучки текста;
- приоритетную очередь;
- расширенную историю;
- доступ к SEO/контент-инструментам.

### SEO Pro

Отдельный тариф или режим внутри Pro, если продукт будет развиваться в сторону SEO и контент-маркетинга.

Может включать:

- генерацию статей;
- контент-планы;
- кластеризацию ключевых слов;
- SEO-заголовки и описания;
- анализ конкурентов;
- массовую генерацию текстов.

## 6. Пример стоимости действий в коинах

Цены ниже являются примером. Их нужно будет настроить после расчета себестоимости.

| Действие | Примерная цена |
| --- | ---: |
| Сообщение в DeepSeek | 1 коин |
| Сообщение в ChatGPT | 3 коина |
| Сообщение в Claude | 2 коина |
| Генерация изображения | 5 коинов |
| Улучшение изображения | 4 коина |
| Генерация короткого видео | 25 коинов |
| Озвучка текста | 5 коинов |
| SEO-статья | 10 коинов |

Важно: пользователь должен видеть не только коины, но и понятное объяснение.

Пример:

`699 рублей = 230 коинов. Этого хватит примерно на 230 сообщений DeepSeek, 76 сообщений ChatGPT или 115 генераций GPT Image.`

## 7. Архитектура MVP

Общая схема:

```text
Telegram Bot
    |
    v
CeaAI Backend
    |
    +--> Database
    |
    +--> Payment Provider
    |
    +--> AI Provider Adapters
    |
    +--> File Storage
```

### Telegram Bot

Отвечает за интерфейс:

- главное меню;
- выбор тарифа;
- покупку подписки;
- выбор нейросети;
- прием запросов пользователя;
- отправку результатов;
- уведомления о балансе и ошибках.

Бот не должен напрямую работать с платежками и AI API. Он общается с backend.

### Backend

Главный сервер CeaAI.

Отвечает за:

- пользователей;
- подписки;
- баланс коинов;
- списания и начисления;
- платежные webhook;
- вызовы AI-провайдеров;
- историю генераций;
- обработку ошибок;
- антифрод и лимиты.

### Database

Хранит:

- пользователей;
- подписки;
- тарифы;
- баланс коинов;
- операции с коинами;
- платежи;
- AI-запросы;
- историю генераций;
- настройки моделей.

Для MVP можно использовать SQLite или PostgreSQL. Для продакшена лучше PostgreSQL.

### Payment Provider

Основной сценарий оплаты:

1. Пользователь выбирает тариф.
2. Backend создает платеж.
3. Платежная система возвращает ссылку.
4. Бот отправляет пользователю кнопку `Оплатить`.
5. Пользователь оплачивает на странице платежки.
6. Платежка отправляет webhook на backend.
7. Backend проверяет подпись webhook.
8. Backend активирует подписку и начисляет коины.
9. Бот уведомляет пользователя об успешной оплате.

Возможные платежки:

- ЮKassa;
- CloudPayments;
- Robokassa;
- Т-Банк Касса.

Важно: начислять коины можно только после подтвержденного webhook от платежной системы.

### AI Provider Adapters

Каждая нейросеть подключается через отдельный адаптер.

Пример:

```text
providers/
  openai
  anthropic
  deepseek
  xai
  image
  video
  tts
```

Backend должен иметь единый внутренний интерфейс:

```text
generateText(model, prompt, userContext)
generateImage(model, prompt, options)
generateVideo(model, prompt, options)
generateSpeech(model, text, options)
```

Так бот не зависит от конкретного API. Если одна модель меняется или отключается, остальная система продолжает работать.

## 8. Как подключаются нейросети

Для каждой нейросети нужны:

- аккаунт разработчика;
- API key;
- понимание тарифов API;
- лимиты;
- формат запроса;
- формат ответа;
- обработка ошибок.

Пример логики:

1. Пользователь выбирает `ChatGPT`.
2. Backend смотрит цену модели в коинах.
3. Backend проверяет баланс.
4. Backend резервирует коины.
5. Backend отправляет запрос в OpenAI API.
6. Если ответ успешный, backend списывает коины окончательно.
7. Если произошла ошибка, backend возвращает коины.
8. Бот отправляет ответ пользователю.

Для долгих задач, например видео и сложной озвучки текста, нужна очередь задач.

Пример:

```text
User request
    |
    v
Create generation job
    |
    v
Резерв коинов
    |
    v
Send job to video/tts provider
    |
    v
Wait via polling or webhook
    |
    v
Save result
    |
    v
Send result to Telegram
```

## 9. Что входит в MVP

MVP должен проверить главный вопрос: готовы ли пользователи платить за удобный AI-агрегатор в Telegram.

Минимальный набор:

- Telegram-бот;
- регистрация пользователя после `/start`;
- главное меню;
- баланс коинов;
- 3 тарифа;
- создание платежной ссылки;
- webhook успешной оплаты;
- начисление коинов;
- текстовая нейросеть;
- генерация изображений;
- история последних генераций;
- админская настройка цен хотя бы через конфиг;
- базовые логи ошибок.

В MVP можно не делать:

- сложную Mini App;
- полноценную админ-панель;
- десятки нейросетей;
- многоуровневую реферальную систему;
- командные аккаунты;
- сложную аналитику;
- SEO Pro-инструменты.

## 12. Сценарий генерации текста

```text
Бот -> "Отправьте запрос. Стоимость: 3 коина."

```

## 15. Правила списания коинов

Для быстрых задач:

1. Проверить баланс.
2. Выполнить запрос.
3. Списать коины после успешного результата.

Для дорогих или долгих задач:

1. Проверить баланс.
2. Зарезервировать коины.
3. Запустить генерацию.
4. Если успешно - списать резерв.
5. Если ошибка - вернуть резерв.

Это важно, чтобы пользователь не терял коины из-за ошибок провайдера.

## 16. Обработка ошибок

Примеры ошибок:

- недостаточно коинов;
- подписка закончилась;
- AI-провайдер недоступен;
- платеж не подтвержден;
- пользователь отправил неподдерживаемый файл;
- генерация заняла слишком много времени.

Бот должен отвечать простым языком.

Пример:

`Не получилось сгенерировать изображение. Коины не списаны. Попробуйте еще раз чуть позже.`

## 19. Главные риски

### Экономика

Если неправильно посчитать стоимость API, можно продавать подписки дешевле себестоимости.

Решение:

- считать примерную себестоимость каждого действия;
- ставить запас по марже;
- быстро менять цены в коинах.

### Стабильность AI API

Некоторые провайдеры могут тормозить, падать или менять цены.

Решение:

- делать адаптеры;
- логировать ошибки;
- возвращать коины при сбоях;
- иметь альтернативные модели.

### Сложность продукта

Если сразу добавить много нейросетей, продукт станет сложным и неудобным.

Решение:

- начать с небольшого набора популярных сценариев;
- сделать понятное меню;
- добавлять модели по спросу.

## 21. Принцип разработки

CeaAI нужно строить не как набор случайно подключенных API, а как платформу.

Правильная логика:

```text
Telegram - это интерфейс.
Backend - это мозг продукта.
AI-провайдеры - это заменяемые модули.
Коины - это внутренняя экономика.
Подписка - это основная модель монетизации.
```

Такой подход позволит сначала быстро запустить Telegram-бота, а потом без полной переделки добавить Mini App, сайт, админку и новые нейросети.
